title: Configuration
description: Choose which .env files load, mirror values back to process.env, and turn on debug logging.
---

envapt reads from `process.env` plus the `.env` files it loads. Static settings on `Envapter` control which files load, whether loaded values reach `process.env`, and how much envapt logs.

```ts twoslash
import { Envapter } from 'envapt';
// ---cut---
Envapter.envPaths = ['.env.local', '.env'];
Envapter.envFileOptions = { override: true };
Envapter.syncProcessEnv = true;
Envapter.debug = 'verbose';
```

<Callout type="info">
    envapt loads `.env` files into an isolated copy of `process.env`, so by default it does not mutate the real
    `process.env`. Values are read from envapt's own cache. `syncProcessEnv` (below) opts into writing them back.
</Callout>

## Which files load

By default envapt loads `.env`, and in a detected environment it layers a per-environment cascade (covered in [Environment](/docs/environment)).

Set `envPaths` to load a fixed, ordered list instead. Earlier paths win.

```ts twoslash
import { Envapter } from 'envapt';
// ---cut---
Envapter.envPaths = ['config/app.env', '.env'];
```

<Callout type="danger">
    Every path you list must exist. Setting `envPaths` validates the paths immediately, and a missing file throws
    `EnvaptError` with code `EnvFilesNotFound`. Setting `envPaths` also takes precedence: it disables the
    per-environment auto-cascade and any `configureProfiles` paths.
</Callout>

`envFileOptions` tunes how files are read. `override` (default `false`) decides whether a later file or a file value can replace an already-set key; `encoding` (default `utf8`) sets the file encoding.

```ts twoslash
import { Envapter } from 'envapt';
// ---cut---
Envapter.envFileOptions = { override: true, encoding: 'utf8' };
```

Use `envPaths` to choose which files load.

## Mirroring to process.env

By default a loaded value lives only in envapt's cache. Set `syncProcessEnv` to mirror the keys envapt loaded back into `process.env`, so other libraries that read `process.env` directly see them.

```ts twoslash
import { Envapter } from 'envapt';
// ---cut---
Envapter.syncProcessEnv = true;
```

Only keys envapt's loader wrote are mirrored, so collisions follow `envFileOptions.override`: with the default `false`, a pre-existing `process.env` value is preserved.

<Callout type="warn">
    Mirroring is one-way. Turning `syncProcessEnv` back to `false` does not remove keys already written; they stay in
    `process.env` until the process exits.
</Callout>

## Drop-in for dotenv

For the `dotenv/config` workflow, import `envapt/config`. It loads the `.env` cascade and mirrors every loaded key into `process.env` in one step, so `process.env.PORT` works without touching `Envapter`.

```ts
import 'envapt/config';

const port = process.env.PORT; // the loaded keys are now on process.env
```

It is the side-effect form of `Envapter.syncProcessEnv = true` plus an eager load, and follows the same collision rules. Unlike `dotenv/config`, it loads the whole per-environment [cascade](/docs/environment), not a single `.env`.

Preload it without editing your entry file:

```bash
node --import envapt/config app.js   # ESM
node -r envapt/config app.js         # CommonJS
```

For typed reads on top, keep using [`Envapter`](/docs/envapter).

## Debug logging

`debug` controls how much envapt writes to stderr (prefixed `[envapt]`). The levels are `silent` (default), `warn`, and `verbose`.

```ts twoslash
import { Envapter } from 'envapt';
// ---cut---
Envapter.debug = 'verbose';
```

- `warn`: signals that may indicate misconfiguration: failed file reads, unresolved templates in non-strict mode, and fallback values used in place of a missing variable.
- `verbose`: adds every loaded file, per-file key counts, per-key load lines, effective paths, and cache rebuilds.

You can also set the level without code through the `ENVAPT_DEBUG` environment variable. envapt reads it on first access; an explicit `Envapter.debug` assignment overrides it.

```dotenv
ENVAPT_DEBUG=verbose
```
