title: Sources

A source is the object envapt reads variables from. On Node, Bun, and Deno one is bound automatically on import, reading `process.env` and the `.env` file cascade. On Cloudflare Workers, in the browser, and on the edge runtimes you bind a source yourself with `Envapter.useSource(...)`, then read with the same typed API.

<Callout type="info">
    On Node, Bun, and Deno you never call `useSource`. The Node entry binds a source that reads a clone of `process.env`
    plus the `.env` cascade at import time.
</Callout>

## The providers

envapt ships two sources.

- `NodeEnvSource` is bound automatically on the Node, Deno, or Bun entry. It reads a clone of `process.env` and the `.env` cascade.
- `PortableSource` wraps any object you pass, for every runtime without a filesystem. Pass the Cloudflare `env` binding on Workers (see [Workers](/docs/workers)), or the config your bundler injects into a browser build like `import.meta.env` (see [Browser](/docs/browser)).

`PortableSource` snapshots the object at construction (so later mutation does not leak in) and JSON-stringifies non-string values so the converters parse them the same way.

<Callout type="warn">
    `ManualEnvSource` and `WorkerEnvSource` are deprecated aliases of `PortableSource`, kept for backward compatibility
    and removed in v8. Replace `new ManualEnvSource(obj)` and `new WorkerEnvSource(env)` with `new PortableSource(...)`.
</Callout>

You pass the source the object the runtime exposes, the Worker `env` binding or the bundler's injected config, not one you write out by hand. The value is the typed read on top:

```ts twoslash
import { Envapter, PortableSource, Converters } from 'envapt';
// ---cut---
declare const env: Record<string, unknown>; // the Cloudflare env binding
Envapter.useSource(new PortableSource(env));

const upstream = Envapter.getUsing('UPSTREAM_URL', Converters.Url); // URL | undefined
const cacheTtl = Envapter.getUsing('CACHE_TTL', Converters.Time, '5m'); // number (ms)
```

## Any object can be a source

`Source` is one required method, `readVars(): Record<string, string>`. Any object that satisfies it works, so you can adapt another config system, a secrets payload or an RPC response, without writing a provider.

```ts twoslash
import { Envapter, Converters } from 'envapt';

import type { Source } from 'envapt';
// ---cut---
declare const secrets: Record<string, string>; // fetched from your secrets manager
const source: Source = { readVars: () => secrets };
Envapter.useSource(source);

const apiBase = Envapter.getUsing('API_BASE_URL', Converters.Url);
const poolSize = Envapter.getNumber('DB_POOL_SIZE', 10);
```

An optional `supportsFiles` flag marks a source as backing the file-only config APIs. Only `NodeEnvSource` sets it, which is why the `.env` cascade and `envPaths` work on Node and nowhere else.

For fetching that secrets object from 1Password, Vault, Doppler, AWS Secrets Manager, and others, see [Using secret stores](/docs/secret-stores).

## When nothing is bound

Reading a value before you call `useSource` throws `EnvaptError` with code `NoSourceBound`. It cannot happen on Node, where the source is bound on import.

The file-only config APIs (`envPaths`, `baseDir`, `envFileOptions`, `configureProfiles`, `resetProfiles`) need a source that backs the filesystem. Calling one on a bare source throws `EnvaptError` with code `FileApiUnsupported`. See [Errors](/docs/errors) for both.

<Callout type="warn">
    Off Node there is no filesystem, so the `.env` cascade and the file-only config APIs are absent. Pass your values
    through `PortableSource` instead.
</Callout>

For the per-runtime flows see [Workers](/docs/workers) and [Browser](/docs/browser). The build selection is in [Compatibility](/docs/compatibility).
